# BLEU-DISTRIBUTION-VALUE-BANDWIDTH
## Source of truth · §9.6 — Six doors, one engine, one ledger
### Signed: Bleu · June 11, 2026 · Banked — distribution architecture, not build scope

**The law:** Every door resolves into the same governed object (`catalyst_event`).
Every surface must create trust, capture consent, classify an open window, route
verified help, or close a return loop. No thin city pages. No app-as-website.
No model-as-magic. No commerce before care. No acquisition that cannot become a
documented catalyst event.

## The surface map

| Surface | Job | Finds | Converts to |
|---|---|---|---|
| bleu.live | public trust home | consumers, caregivers, partners | start-here / trust / explanation |
| cities.bleu.live | civic trust packets | residents, navigators, local-intent search | local support check, text opt-in |
| enterprises.bleu.live | buyer engine | hospitals, FQHCs, employers, cities, payers | pilot request → contract |
| App (BLEU Passport) | retention vessel | people needing ongoing support | passport, care circle, reminders, consent |
| SMS | active contact spine | people already inside a window | route + return loop |
| Model layer | invisible intelligence | every surface | siren, rail, routing, rationale, audit |
| Google / AI search | discovery | people searching pain, not brand | page → local check → intake |

## Surface laws (each inherits the six non-negotiables)
- **cities.bleu.live:** a city page earns indexation only when it can help a human
  in that city today — verified resource density, last-verified dates, source links,
  no-match honesty, schema markup only where visible content supports it. No mass
  thin templates (doorway-page risk). Canonical: cities.bleu.live; any citys.*
  variant is redirect-only. This is also the remediation frame for the templated
  city-page attribution issue.
- **enterprises.bleu.live:** verticalized buyer pages (rural hospital, FQHC/RHC,
  employer, city, Medicaid/ACO, reentry). CTA: "Show us one handoff you are
  accountable for. We will show you who is still in the open window."
- **App:** BLEU Passport — after-the-handoff companion. No diagnosis, no dosing,
  no PHI in push, no forced location, consent controls, app-to-SMS continuity.
  App Review reality: medical-adjacent apps face scrutiny; the app is the retention
  vessel, never "the AI doctor."
- **Model layer:** role-separated (intake / siren / rail / routing / rationale /
  audit / content / commerce). The model is not the moat; the governed event
  network is. Cheapest safe model for ordinary classification, strongest governed
  model for ambiguity and safety; no model may invent a resource, clinical claim,
  or billing claim. Tier decision = human call, pending.

## Acquisition loops
1. Google intent ("I got discharged and don't understand my medicine") → Learn/city
   page → "what happened, where are you, want a text reminder?" → intake.
2. City trust loop — navigators, librarians, pastors, city staff using the packet.
3. Enterprise enrollment — QR on discharge sheet / benefits link / case-manager
   invite (highest value; institution already accountable).
4. App Store → Passport → SMS consent → return loop.
5. SMS — the active rescue channel; D+28 reply is proof-of-life.
6. Partner API (future): classifyWindow / assignSiren / assignRail / resolveRoute /
   writeRationale / scheduleReturn / closeWindow.

## Value bandwidth (largest first)
Recurring institutional infrastructure revenue attached to accountable handoffs
(PMPM, setup, per-episode, dashboards, payer reporting) → civic dashboards →
app retention → governed data intelligence (care-desert maps, no-match geography)
→ eventual API licensing. Affiliate commerce: minor, gated, never the lead.

> Google finds the question. Cities prove the local truth. Enterprises pay for
> accountability. The app holds the passport. SMS closes the window. The models
> route the risk. The ledger becomes the network.
