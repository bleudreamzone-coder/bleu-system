NOTE on conversation_history (RLS on, 0 policies): the table has RLS
enabled but no CREATE POLICY in the dump. With RLS on and zero
policies, the default deny applies to all non-service-role roles. The
server.js path uses the service-role key (which bypasses RLS), so
backend reads/writes still work. But the frontend anon key has zero
read access. This may be intentional (anon should not read history
directly) or a missing policy that breaks a UX path. Worth confirming.

=====================================================================
GRANT/REVOKE STATE FOR THE 7 P0 TABLES
=====================================================================

Every P0 table has:
  GRANT ALL ON TABLE ... TO "anon";
  GRANT ALL ON TABLE ... TO "authenticated";
  GRANT ALL ON TABLE ... TO "service_role";

There is no REVOKE for any of them. Combined with RLS off, the access
posture is fully open to anyone holding the anon JWT — which is
public by design (embedded in index.html line 795).

Attack surface check:
  curl 'https://sqyzboesdpdussiwqpzk.supabase.co/rest/v1/user_coherence?select=*' \
       -H "apikey: <anon-jwt-from-index.html>" \
       -H "Authorization: Bearer <anon-jwt-from-index.html>"
  → would dump every row of user_coherence (every user's phone, CI, ISI)

The default privileges line:
  ALTER DEFAULT PRIVILEGES FOR ROLE "postgres" IN SCHEMA "public"
    GRANT ALL ON TABLES TO "anon";

means any NEW table created in this schema also inherits anon-GRANT
ALL automatically. This is the root cause for the broader "28 tables
without RLS" picture and needs to be addressed at the schema-default
level, not table-by-table.

=====================================================================
RECOMMENDED P0 FIX SEQUENCE (for Bleu to choose path)
=====================================================================

OPTION A — Minimum patch, ship today
  In Supabase SQL editor, for each of the 7 user-data tables:
    ALTER TABLE public.<t> ENABLE ROW LEVEL SECURITY;
    CREATE POLICY "<t>_own_row" ON public.<t>
      FOR ALL TO authenticated
      USING (user_id = auth.uid())
      WITH CHECK (user_id = auth.uid());
    REVOKE ALL ON public.<t> FROM anon;
  Then re-run this audit. Single migration file. ~30 min.

OPTION B — Full hardening
  Option A + revoke default privileges to anon at the schema level:
    ALTER DEFAULT PRIVILEGES IN SCHEMA public REVOKE ALL ON TABLES FROM anon;
    REVOKE ALL ON ALL TABLES IN SCHEMA public FROM anon;
    Then explicitly GRANT SELECT on the read-only tables (cities,
    locations, products, protocols, etc.) that the frontend actually
    needs to read with the anon key.
  Higher friction (catalog of what the frontend actually reads with
  anon must be built first). Right thing long-term.

OPTION C — Service-role-only enforcement
  If the 7 P0 tables are written ONLY by server.js (service role),
  not by the frontend with the anon key:
    REVOKE ALL ON public.<t> FROM anon, authenticated;
  No RLS needed because no non-service role can touch the table.
  Confirm by grep — see "Verification before fix" below.

Verification before fix (which of the 7 tables does the frontend
write to via the anon key vs. server.js write via service role?):

  Frontend anon-key writes (per audit 09):
    profiles, conversations, product_feedback, session_feedback,
    user_signals (last three don't exist in this dump)

  So none of the 7 P0 tables are in the documented frontend-anon
  write list. They are likely all server-write only (server.js or
  edge functions using service role). That means OPTION C is
  probably the right answer for most of them. user_coherence is
  the one to confirm — the SMS reorder path writes phone numbers
  somewhere, and if that's a frontend anon write, the policy needs
  to allow that specific path.
