=====================================================================
OPEN QUESTION (defense-in-depth) — separate follow-up
=====================================================================

The 7 P0 tables now have only `service_role` grants. They are inaccessible
to anon and authenticated. The exposure is closed.

However, RLS itself is still OFF on those 7 tables. Per Option C from
the morning audit, REVOKE-only is equivalent in posture to RLS-on-with-
zero-policies when no non-service role has any GRANT — and it is.

If a future migration accidentally re-grants anon (e.g. via
`GRANT SELECT ON ALL TABLES IN SCHEMA public TO anon`), the REVOKE
posture would silently break and the tables would be re-exposed. Enabling
RLS on these 7 tables — even with zero policies — would prevent that
class of regression because RLS-on-with-no-policy is default-deny for
non-service roles.

This is the optional "defense-in-depth" block from the migration file's
comments (lines 43-55). Not in scope for tonight's session. Logged for
a future follow-up (FU-007? — to be assigned when added to followups/).
